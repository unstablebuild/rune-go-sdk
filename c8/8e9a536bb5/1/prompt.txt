# Minimal Darwin Sandbox with Embedded Images

## Context

The current sandbox implementation depends on:
- **Matchlock** - Go wrapper around Apple's Virtualization.framework
- **e2fsprogs** - Required at runtime for ext4 image manipulation
- **Downloaded binaries** - guest-agent and guest-fused from GitHub

These external dependencies complicate deployment and make the package harder to use. The goal is to create a minimal implementation that:
1. Uses `github.com/Code-Hex/vz` directly (pure Go bindings)
2. Has **zero runtime dependencies** (no e2fsprogs, no downloads)
3. Embeds a pre-built Alpine Linux kernel and rootfs
4. Supports darwin/arm64 only (Apple Silicon)

### 5. VM Configuration Details

Using Code-Hex/vz v3 APIs:

```go
// Boot loader
bootLoader, _ := vz.NewLinuxBootLoader(
    kernelPath,
    vz.WithCommandLine("console=hvc0 root=/dev/vda rw init=/init"),
)

// VM config
vmConfig, _ := vz.NewVirtualMachineConfiguration(bootLoader, cpus, memoryBytes)

// Storage - rootfs as virtio block device
diskAttachment, _ := vz.NewDiskImageStorageDeviceAttachment(rootfsPath, false)
storageConfig, _ := vz.NewVirtioBlockDeviceConfiguration(diskAttachment)
vmConfig.SetStorageDevicesVirtualMachineConfiguration(...)

// Network - NAT for internet access (filtered by guest-side iptables)
natAttachment, _ := vz.NewNATNetworkDeviceAttachment()
netConfig, _ := vz.NewVirtioNetworkDeviceConfiguration(natAttachment)
vmConfig.SetNetworkDevicesVirtualMachineConfiguration(...)

// Vsock - for host-guest communication
socketConfig, _ := vz.NewVirtioSocketDeviceConfiguration()
vmConfig.SetSocketDevicesVirtualMachineConfiguration(...)

// Shared directory - workspace mounting via virtio-fs
sharedDir, _ := vz.NewSharedDirectory(config.WorkspaceRoot, readOnly)
share, _ := vz.NewSingleDirectoryShare(sharedDir)
fsConfig, _ := vz.NewVirtioFileSystemDeviceConfiguration("workspace")
fsConfig.SetDirectoryShare(share)
vmConfig.SetDirectorySharingDevicesVirtualMachineConfiguration(...)
```

### 6. Guest Init Script (`init.sh`)

```sh
#!/bin/sh
# Minimal init for sandbox VM

# Mount essential filesystems
mount -t proc proc /proc
mount -t sysfs sysfs /sys
mount -t devtmpfs devtmpfs /dev
mount -t devpts devpts /dev/pts
mount -t tmpfs tmpfs /run
mount -t tmpfs tmpfs /tmp

# Mount workspace via virtio-fs
mkdir -p /workspace
mount -t virtiofs workspace /workspace 2>/dev/null || true

# Configure network
ip link set lo up
ip link set eth0 up
udhcpc -i eth0 -s /etc/udhcpc.script -q -n 2>/dev/null || true

# Start guest agent as PID 1
exec /usr/bin/guest-agent
```

### 7. Build Process

**Prerequisites (build-time only, NOT runtime):**
- Docker (for rootfs building)
- Go (for cross-compiling guest-agent)

**Build script (`scripts/build-guest-images.sh`):**

1. **Build guest-agent** (cross-compile):
   ```bash
   GOOS=linux GOARCH=arm64 CGO_ENABLED=0 go build -ldflags="-s -w" \
       -o images/guest-agent ./sandbox/guest/
   ```

2. **Build rootfs** (Docker-based, no e2fsprogs on host):
   ```dockerfile
   FROM alpine:3.19 as rootfs
   RUN apk add --no-cache ca-certificates
   COPY guest-agent /usr/bin/guest-agent
   COPY init.sh /init
   RUN chmod +x /init /usr/bin/guest-agent

   FROM scratch as export
   COPY --from=rootfs / /rootfs
   ```

   Then create ext4 image inside Docker:
   ```bash
   docker run --rm -v $(pwd):/out alpine sh -c "
     apk add e2fsprogs
     dd if=/dev/zero of=/out/rootfs.ext4 bs=1M count=100
     mkfs.ext4 /out/rootfs.ext4
     mount /out/rootfs.ext4 /mnt
     cp -a /rootfs/* /mnt/
     umount /mnt
   "
   zstd -19 rootfs.ext4 -o images/rootfs/rootfs-arm64.ext4.zst
   ```

3. **Download/build kernel**:
   - Option A: Use pre-built kernel from Alpine/Linuxkit
   - Option B: Build minimal kernel with custom config
   ```bash
   curl -L https://boot.alpinelinux.org/kernels/arm64/vmlinuz-lts -o images/kernel/vmlinuz-arm64
   gzip images/kernel/vmlinuz-arm64
   ```

### 8. Network Policy Enforcement

Since we use NAT networking, additional filtering via guest-side iptables:

```sh
# In init.sh, if network policy is restrictive
if [ -f /etc/sandbox/network-policy ]; then
    iptables -P OUTPUT DROP
    iptables -A OUTPUT -m state --state ESTABLISHED,RELATED -j ACCEPT
    # Allow specific hosts from policy file
    while read host; do
        iptables -A OUTPUT -d "$host" -j ACCEPT
    done < /etc/sandbox/allowed-hosts
fi
```

### 9. Security Features (Process Sandboxing Level)

Guest-side security (already in sandboxrpc.Server):
- Commands execute in their own PID/mount namespaces (optional enhancement)
- Working directory defaults to /workspace
- Environment scrubbed by host-side policy before reaching guest

VM-level isolation:
- Separate VM per sandbox
- No shared state between VMs
- Network isolated via NAT
- Filesystem isolated except explicit workspace mount
